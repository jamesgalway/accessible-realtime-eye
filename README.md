# 视障导航网页测试原型

结论：这个项目不是完整导航 App，而是一个能先跑起来的网页端 MVP，用来验证“地图导航状态 + 实时慧眼确认 + 语音可执行指令”这条链路。

## 当前能力

- 用高德 Web 服务搜索 POI。
- 用高德 Web 服务规划公交加步行或纯步行路线。
- 后端串行调用高德接口，避免测试时并发请求混乱。
- 手机网页打开摄像头和麦克风。
- 后端支持百炼实时 WebSocket 代理，前端可做“类似视频通话”的低频画面和语音测试。
- 如果部署平台没有读到服务端百炼 Key，手机页面可以临时填写 DashScope Key 做实时测试。
- 如果没有配置高德 Key，也可以手工导入路线事实，先测试到达区域的实时慧眼工作流。

## 运行方法

1. 进入项目目录。

```powershell
cd E:\accessible-nav-prototype
```

2. 复制配置文件。

```powershell
Copy-Item .env.example .env
```

3. 编辑 `.env`，填入自己的高德 Web 服务 Key 和百炼 Key。

```text
AMAP_KEY=你的高德Key
BAILIAN_API_KEY=你的百炼或DashScope Key
```

实时模型测试还需要配置：

```text
DASHSCOPE_API_KEY=你的DashScope Key
BAILIAN_WEBRTC_ENDPOINT=阿里百炼控制台给出的WebRTC endpoint
```

如果只是手机公网测试，也可以不把百炼 Key 放进部署平台后台。打开网页后，在“视频通话式慧眼确认”区域填写 DashScope Key，点“保存到本机浏览器”，再启动实时慧眼。这个 Key 只保存在当前手机浏览器本地，后端只在本次连接时临时使用，不写入 GitHub 或 Render。

4. 启动本地服务。

```powershell
npm.cmd start
```

5. 在浏览器打开本机地址。

```text
http://localhost:8787
```

## 手机测试注意

- 电脑和手机在同一个局域网时，可以用电脑局域网 IP 加端口访问。
- iPhone Safari 对摄像头和定位权限比较严格。不是 localhost 时，通常需要 HTTPS，局域网 HTTP 可能不能打开摄像头。
- iPhone 上建议使用 HTTPS 公网地址测试摄像头和麦克风权限。
- GitHub Pages 只能放静态页面，不能安全保存 Key 或代理实时 WebSocket；真正可用的手机测试版需要部署到支持 Node 后端的平台。

## 部署说明

- GitHub 仓库只保存源码和部署配置，不保存 `.env`。
- `render.yaml` 用于后续接 Render 这类 Node 托管平台。
- Cloudflare Workers 测试时出现 workers.dev 1101，已停止继续排查，避免在错误方向消耗时间。
- Render 环境变量如果没有生效，不影响实时测试；用手机页面临时填写 DashScope Key 即可绕过部署平台变量问题。

## 安全和开源注意

- `.env` 已经被 `.gitignore` 排除，不能把真实 Key 放进公开仓库。
- 高德 Key 仍只在后端使用。百炼 Key 支持两种方式：服务端环境变量，或测试手机临时填写后只在本次实时连接中传给后端代理。
- 开源时保留 `.env.example`，不要提交 `.env`。

## 重要限制

- 高德原始步行指令不等于最终给视障者执行的“左边、右边、过几次马路”导航稿。
- 浏览器定位坐标和高德坐标可能不是同一坐标系，精确判断路侧前必须做坐标校准。
- 目的地 POI、路边导航落点、公交站名、站杆位置、摄像头看到的入口必须分开保存，不能混成一个点。
- 如果地图和摄像头识别冲突，系统必须提示冲突，不能编确定结论。

## 后续适合做的版本

- 把高德返回的路线拆成“每个步行段的结构化事实”。
- 增加坐标统一和左右侧物理推理模块。
- 增加“到达区域辅助模式”：到终点附近后不结束导航，而是继续帮用户找门牌、站牌、入口。
- 增加 HTTPS 本地测试方案，方便 iPhone 直接打开摄像头。
- 继续完善百炼实时音视频模型，把“到达区域辅助模式”做成可长期测试的手机网页。
