# 导航和慧眼模式的工作流说明

结论：我们要验证的不是“地图界面和视频通话界面同时显示”，而是把它们改成同一个连续出行任务里的两个能力。

## 核心设计

导航负责主路径：起点、终点、公交或步行方案、下一段路、预计道路、预计目标侧。

摄像头确认负责地图不擅长的现场问题：公交站牌、门牌号、医院入口、商铺招牌、是否已经到门口、现场目标在左前方还是右前方。

整个过程必须保留同一个导航状态。用户打开摄像头时，模型不能只看画面泛泛描述，而要带着导航上下文回答。

## 状态必须分开保存

- 目的地 POI 点：地图里这个地点的真实位置。
- 路边导航落点：路线规划把人带到哪一段路边。
- 公交站名：例如某某路某某路，只是名字，不一定等于站杆。
- 公交站杆点：真正上车或下车的路边位置。
- 摄像头看到的目标：现场画面里的站牌、门牌、入口或招牌。

这几个点不能混用。之前“洛川路阳光201”和“1208路车站侧边”的错误，本质都是把不同点混成一个点。

## 原型流程

1. 用户输入起点和终点。
2. 后端调用高德搜索 POI，先锁定地图事实。
3. 后端调用高德规划路线，形成路线骨架。
4. 前端把路线用大字和语音读出来。
5. 用户接近公交站、路口或目的地时，打开摄像头。
6. 前端打开类似视频通话的实时模式，把低频视频帧、语音和当前导航状态一起发给后端。
7. 后端通过百炼实时模型代理连接，持续传递现场确认问题和用户语音。
8. 模型只回答现场确认问题，例如是否看到目标、目标方向、下一步行动、不确定点。
9. 如果地图和图片冲突，必须提示冲突，不能强行确认。

## 给智能眼镜厂家的借鉴

眼镜不一定要让“导航模式”和“慧眼模式”两个界面共存。更现实的做法是做一个导航辅助慧眼层。

导航任务在后台持续存在，里面保存目的地、当前路段、下一步动作、目标可能在左侧还是右侧、需要找的现场目标。

用户说“帮我看一下入口”或“帮我看站牌”时，慧眼短暂接管摄像头，但它拿到的是带导航上下文的任务，而不是一次孤立的拍照识别。

识别完成后，系统回到导航任务，并播报可执行结果，例如“疑似看到医院门诊入口，在右前方，先不要过马路，原地慢慢向右扫视确认门牌”。

## 当前实时方案

当前版本优先使用 WebSocket 实时流：手机打开摄像头和麦克风，前端按低频抽帧发送画面，同时发送语音和导航上下文。这样更接近慧眼视频通话，但成本和实现复杂度低于完整 WebRTC。

如果部署平台没有正确读到百炼 Key，手机页面可以临时填写 DashScope Key。Key 只保存在手机浏览器本地，后端只在本次连接时临时使用，不写入仓库或部署配置。

WebRTC 通话仍作为后续方向保留，需要百炼白名单 Endpoint 稳定可用后再做深度测试。

## 开源边界

可以开源网页、后端、状态格式和工作流规则。

不能开源真实 API Key、用户家庭地址、常用路线隐私数据、未脱敏的图片和日志。
